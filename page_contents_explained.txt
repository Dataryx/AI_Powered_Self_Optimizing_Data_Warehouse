ROUTE: /  —  Data Warehouse Dashboard (DashboardPage)
================================================================================

Overall purpose
--------------------------------------------------------------------------------
This is the landing experience: prove the warehouse is loaded, the API answers,
and Gold-layer analytics look credible. I poll the same bundle on an interval
from user preferences.

Block 1 — TopBar
--------------------------------------------------------------------------------
I show navigation links to the rest of the app, a live or offline indicator,
local time, and quick access to alerts. The bar receives the same dashboard data
hook so it can show connection health and a compact list of active alerts with
severity coloring. If loading fails, I surface connectionError so you see
Offline instead of a silent failure.

Block 2 — HeroSection
--------------------------------------------------------------------------------
This is decorative framing, not database-backed: badges for Overview, three
layers, and Online; a large title Data Warehouse Dashboard; and light SVG
contour lines. It sets the visual tone before any numbers appear.

Block 3 — Error banner (conditional)
--------------------------------------------------------------------------------
If useDashboardData sets an error string, I print it above the metrics with a
Retry button. I still attempt to show fallback content where partial data exists.

Block 4 — KeyMetrics (Section 01 — Key Statistics)
--------------------------------------------------------------------------------
I render six cards from the bundled summary and sales payloads:
  Total Records — sum of estimated rows across Bronze, Silver, and Gold from
    warehouse_summary.
  Total Tables — sum of table counts across the three layers.
  Total Sales — transaction count from total_sales.count when present.
  Total Revenue — formatted dollars from total_sales.revenue.
  Avg Sale Value — derived from revenue and count when both exist.
  Total Customers — from customer stats when the API returns it.

Block 5 — MedallionTiers (Section 02 — Medallion Architecture)
--------------------------------------------------------------------------------
Three columns for Bronze, Silver, and Gold: table count, estimated row count,
and short descriptions (raw ingestion, cleaned and conformed, business-ready).
If the API has not returned warehouse_summary yet, I show placeholder tier
numbers so the layout never collapses.

Block 6 — SalesTrend
--------------------------------------------------------------------------------
A wider chart area (three columns on XL grid) showing daily sales from the sales
payload: dates with counts or revenue depending on what I receive. This ties
the story to time-series behavior in Gold.

Block 7 — TopProducts
--------------------------------------------------------------------------------
A narrower column beside the trend listing top products by revenue or sales
counts from the same sales statistics object.

Block 8 — Footer
--------------------------------------------------------------------------------
Standard footer content for the shell page.

ROUTE: /monitoring  —  ETL Monitoring (MonitoringPage)
================================================================================

Overall purpose
--------------------------------------------------------------------------------
I use this page to defend pipeline reliability: jobs, lineage, throughput,
freshness against expectations, and quality checks. Hash fragments like
#monitoring-lineage scroll to a section after navigation.

Shell — MonitoringHeader
--------------------------------------------------------------------------------
I include a compact header component for the monitoring context (refresh affordance
and title styling consistent with the page).

Block 1 — Page title row
--------------------------------------------------------------------------------
Satellite icon, title ETL Monitoring, subtitle about real-time pipeline health
and freshness.

Block 2 — Error banner (conditional)
--------------------------------------------------------------------------------
Same pattern as other pages: message plus Retry calling refetch from
useMonitoringData.

Block 3 — ETLStats (stats row)
--------------------------------------------------------------------------------
Five summary tiles I derive from the monitoring bundle plus anomalies:
  Active Pipelines — count of pipeline activity I compute from job metadata.
  Failed Runs — all-time failed or error runs from failedRunsTotal on the API,
    not from the length of the mixed errors list.
  Data Freshness SLA — fraction of tables meeting freshness expectations versus
    scanned, using the freshness payload.
  Avg ETL Duration — average duration with a delta versus baseline when I have
    throughput timing data.
  ML-Detected Anomalies — count aligned with GET /alerts/anomalies so this row
    matches the Alerts page notion of anomaly volume.

Block 4 — LineageVisualization (id monitoring-lineage)
--------------------------------------------------------------------------------
I render the pipeline DAG object from the API: how Bronze, Silver, and Gold
jobs connect logically. This supports my architecture narrative.

Block 5 — RecentETLRuns and ManualETLJobRunner (grid, id monitoring-etl)
--------------------------------------------------------------------------------
On the left I list recent runs with status and timing. On the right I expose
approved job names from definitions so an operator can POST a manual run and
then refetch. I pass onAfterDispatch to refresh state after a trigger.

Block 6 — DataFreshness (id monitoring-freshness)
--------------------------------------------------------------------------------
Per-dataset or per-table freshness, SLA-style buckets such as on time, at risk,
and breach, plus actions to refresh from the server. This is where I connect ETL
completion times to business expectations.

Block 7 — DataQuality (id monitoring-data-quality)
--------------------------------------------------------------------------------
Layered quality metrics: null rates, rule outcomes, and summaries I receive from
GET /monitoring/data-quality (or embedded in the monitoring-page bundle). I use
it to argue that Silver and Gold conform to constraints.

ROUTE: /data-explorer  —  Data Explorer (DataExplorerPage)
================================================================================

Overall purpose
--------------------------------------------------------------------------------
Physical proof of the medallion model: which tables exist, how large they are,
and exact column types. I start from a static table list so the grid is never
empty, then replace it with live schema discovery.

Shell — Sticky header
--------------------------------------------------------------------------------
Mobile menu, back to Home, breadcrumb Data Explorer, Live badge, and local time
that updates every second.

Block 1 — Title
--------------------------------------------------------------------------------
Heading Data Explorer and one-line description about browsing schemas across
layers.

Block 2 — Layer tabs
--------------------------------------------------------------------------------
Three tabs: Bronze, Silver, Gold. Each shows a count of tables discovered for
that layer. I also print total tables across layers.

Block 3 — Search field
--------------------------------------------------------------------------------
Filters table names within the active layer; Clear resets.

Block 4 — Results caption
--------------------------------------------------------------------------------
Shows how many tables match in the selected layer while loading or after load.

Block 5 — Table cards grid
--------------------------------------------------------------------------------
Each card: table name, layer badge, column count (from stats when available),
and a visual density bar. Click opens the detail modal.

Block 6 — Empty state
--------------------------------------------------------------------------------
If search filters everything out, I show a friendly empty state with clear search.

Block 7 — Footer legend
--------------------------------------------------------------------------------
Dots per layer with counts and total tables.

Block 8 — Detail modal (on card click)
--------------------------------------------------------------------------------
Full-screen overlay with qualified name schema.table, copy button, overview
metrics (exact row count, live row estimate, columns, total size in bytes),
and a scrollable column definition table from limit-zero data fetch.

ROUTE: /optimizations  —  Performance Optimization Center (OptimizationsPage)
================================================================================

Overall purpose
--------------------------------------------------------------------------------
I consolidate recommendations, historical applies, and query-performance analysis
in one place. The header shows whether WebSocket streaming, HTTP polling, or
offline mode is active.

Shell — Header
--------------------------------------------------------------------------------
Back to Home, title breadcrumb Optimizations, stream status pill (Live, Polling,
Connecting, or Offline), local clock.

Block 1 — Title block
--------------------------------------------------------------------------------
Large title Performance Optimization Center; duplicate stream pill, last updated
time, and manual refresh icon that bumps refreshKey in my hooks.

Block 2 — KPI strip (three cards)
--------------------------------------------------------------------------------
  Recommendations — total count with subtext splitting index versus partition
    recommendations from the pending list.
  Slow queries — count of rows returned for the selected performance window,
    with note that sorting follows total run time in the period.
  Update mode — Live versus Polling versus Idle based on WebSocket state so you
    can see whether recommendations stream or fall back.

Block 3 — Error banner (conditional)
--------------------------------------------------------------------------------
Stream or bundle errors with Retry.

Block 4 — Recommendations panel (tabbed)
--------------------------------------------------------------------------------
Outer card titled Recommendations with inner tabs Index and Partition showing
counts. Index tab renders IndexRecommendations; Partition tab renders
PartitionRecommendations. Each row typically includes rationale, target objects,
confidence or scores where I surface them, and Implement when policy allows.

Block 5 — Query Performance Analysis
--------------------------------------------------------------------------------
QueryPerformance wraps the REST-driven slow-query table for the date range the
user selects (Last 7, 30, or 90 days). I pass the UTC window string for transparency
and separate loading or error state from the WebSocket path.

Block 6 — Optimization History
--------------------------------------------------------------------------------
OptimizationHistory lists past recommendation events (applied, failed, pending
history) from the same data object as the stream.

ROUTE: /analytics  —  Analytics (AnalyticsPage)
================================================================================

Overall purpose
--------------------------------------------------------------------------------
Analyst-grade view: aggregates, ML-assisted workload and cache insight, hotspot
trends, slow-query impact, and busy-hour patterns. Jump links in the header scroll
to sections.

Shell — Header
--------------------------------------------------------------------------------
Back to home, title Analytics, last updated timestamp, Refresh button that runs
both analytics and workload-cache insights.

Block 1 — Title and section navigation
--------------------------------------------------------------------------------
Heading Analytics; short tagline; buttons for Overview, Smart insights, ML hotspots,
Slow queries, Busy times that scroll to element ids.

Block 2 — Core error alert (conditional)
--------------------------------------------------------------------------------
If useAnalyticsData fails entirely, I show an error panel with Try again.

Block 3 — Degraded mode banner (conditional)
--------------------------------------------------------------------------------
If the bundle is degraded or storage or cost aux calls failed, I explain which
parts are stale so I do not overclaim accuracy.

Block 4 — AnalyticsStats (overview id analytics-overview)
--------------------------------------------------------------------------------
Top-line stats derived from query performance slices and rollups: for example
queries run with one-day versus seven-day call totals when rollups exist, slow
query rate, typical wait time, and peak busy hour (preferring database hourly
series when provided).

Block 5 — Smart insights row (id analytics-ml-workload-cache)
--------------------------------------------------------------------------------
Two-column grid:
  WorkloadCacheMlPanels — workload cluster distribution and profiles from
    GET /optimization/workload-clusters, with drill-down pagination per cluster
    via GET /optimization/workload-clusters/{id}/queries.
  MLHotspotTimeline — deterministic ranking of queries whose recent latency
    diverges from longer windows; not a separate neural network.

Block 6 — Slow queries and busy times (grid)
--------------------------------------------------------------------------------
  QueryPerformanceImpact — multi-window slow query table with impact columns.
  WorkloadPatterns — hourly or rollup chart for busy times using hourlyCallsUtc7d
    and related fields, respecting metrics aggregation preference.

ROUTE: /alerts  —  Alerts (AlertsPage)
================================================================================

Overall purpose
--------------------------------------------------------------------------------
Operational inbox: what is firing now, what the anomaly detector and heuristics
found, and incident records. Hash routing can select inbox, anomalies, or
incidents tabs.

Shell — Header
--------------------------------------------------------------------------------
Navigation home, Alerts title, refresh, clock.

Block 1 — Tab bar
--------------------------------------------------------------------------------
Three tabs: Inbox, Anomalies, Incidents. I style severity consistently with icons.

Block 2 — Summary meta cards (when data loads)
--------------------------------------------------------------------------------
Optional chips or counts for totals by severity or type from the bundle meta.

Block 3 — Inbox tab
--------------------------------------------------------------------------------
Paginated list of active alerts (page size five in code). Each row shows severity,
title, time; expand or open detail with structured fields such as alert id,
type, layer, table, acknowledgment state. I support batch acknowledge where the API
allows.

Block 4 — Anomalies tab
--------------------------------------------------------------------------------
Paginated anomaly cards including model-driven rows (Isolation Forest on query
logs) and heuristic rows (ETL failures, insert-rate drops, unusual sizes). Detail
panes show actual versus expected when those fields exist.

Block 5 — Incidents tab
--------------------------------------------------------------------------------
Paginated incidents with status open or resolved, timestamps, related alert
counts, affected tables when provided.

Block 6 — Pagination controls
--------------------------------------------------------------------------------
Per-tab first, previous, numbered pages, next, last with range text such as
showing one through five of total.

ROUTE: /settings  —  Settings (SettingsPage)
================================================================================

Overall purpose
--------------------------------------------------------------------------------
Split controls: server-backed alert rules versus local-only preferences stored in
the browser.

Shell — Header
--------------------------------------------------------------------------------
Home link, Settings title, Live badge, clock.

Block 1 — Intro copy
--------------------------------------------------------------------------------
One sentence: tune alert rules on the API and local preferences.

Block 2 — Tab list
--------------------------------------------------------------------------------
  Alert rules — subtitle Server-side thresholds; opens AlertSettings which reads
    and posts GET and POST /alerts/config through api.ts.
  Preferences — subtitle Local feature flags; opens SystemSettings for monitoring
    refresh intervals, retention days, aggregation mode, and alert poll interval
    persisted in monitoringPreferences.

Block 3 — AlertSettings (when alerts tab active)
--------------------------------------------------------------------------------
Per alert type: enable flag, severity, threshold where applicable; batch save to
the API.

Block 4 — SystemSettings (when system tab active)
--------------------------------------------------------------------------------
Sliders or selects that only affect this browser unless I later sync them; they
drive hooks such as useRetentionDays and poll intervals on Analytics, Monitoring,
Alerts, and the home dashboard.
