SLIDE 2 — PROJECT OVERVIEW
────────────────────────────────────────────────────────────────────────────────
  • I will begin with a brief overview of what I built.
  • The central idea is a hybrid machine-learning framework for telemetry-driven optimization of PostgreSQL in near real time.
  • I integrated four models. XGBoost performs supervised regression on mean execution time in milliseconds, which I use to prioritize slow query fingerprints.
  • K-Means partitions the workload into five clusters so that different behavioral groups can be discussed separately.
  • Isolation Forest provides unsupervised anomaly detection on the engineered features, which reduces reliance on hand-labeled anomaly examples.
  • Random Forest performs binary classification to estimate whether cache-oriented tuning is likely to be worthwhile for a given class of queries.
  • The implementation uses FastAPI — fifty-one REST endpoints — together with WebSocket streaming and a React eighteen dashboard with seven lazy-loaded routes.
  • My own contribution includes the dataset pipeline, model training and evaluation, API integration, dashboard development, and reproducibility scripts for the repository.

SLIDE 4 — PROBLEM CONTEXT
────────────────────────────────────────────────────────────────────────────────
  • I would like to motivate why this project matters before describing the solution.
  • PostgreSQL operates at very large scale; the slide cites industry context — on the order of forty-one zettabytes under management by twenty twenty-four and strong annual growth — which underscores increasing tuning pressure.
  • Traditional tools such as pgAdmin, pgBadger, auto explain, and commercial advisors often depend on fixed patterns and thresholds.
  • One limitation is pattern dependence: when applications issue novel SQL shapes, rule-based checks may fail silently.
  • The slide cites empirical noise rates on the order of thirty-five to sixty percent in some advisor studies, which strains operator time.
  • Another limitation is that manual thresholds rarely scale across mixed business-intelligence, ETL, and ad hoc workloads on a single cluster.
  • Literature summarized on the slide also notes that administrators may leave a substantial fraction of throughput unrealized compared to stronger automated approaches.
  • In my work I therefore combine supervised regression, unsupervised anomaly scoring, and clustering, and I route model outputs through a validation layer rather than presenting raw scores alone.

SLIDE 5 — RESEARCH APPROACH (PART 1)
────────────────────────────────────────────────────────────────────────────────
  • I assigned each model a single responsibility so that results remain interpretable for evaluation and for the thesis document.
  • XGBoost predicts mean_exec_time_ms; the presentation material targets at least ninety percent R-squared on held-out telemetry.
  • I selected gradient-boosted trees because they perform well on tabular features, capture nonlinear relationships between SQL structure and runtime, and yield feature importances that I surface in the dashboard for explainability.
  • Isolation Forest operates on the same feature vector as regression, which keeps anomaly detection aligned with the latency model’s geometry.
  • I map isolation scores into severity levels for the user interface.

SLIDE 6 — RESEARCH APPROACH (PART 2)
────────────────────────────────────────────────────────────────────────────────
  • The remaining two models characterize workload type and cache opportunity.
  • K-Means segments traffic into five clusters, as indicated on the slide — sufficient separation without an excessive number of labels for a committee-style operator.
  • Centroid inspection helps me describe what each cluster represents in plain language.
  • Random Forest performs binary classification for cache-beneficial versus not, trained from buffer-related signals in my pipeline.
  • The classifier’s probabilities support ranking of cache-oriented recommendations.
  • Taken together, the four models form the hybrid multi-task design I defend in this seminar: latency prediction, anomaly detection, segmentation, and cache classification, followed by fusion and catalog validation.

SLIDE 8 — SYSTEM ARCHITECTURE
────────────────────────────────────────────────────────────────────────────────
  • I will now summarize the system architecture I implemented.
  • The client tier uses React eighteen with Vite and TypeScript — seven routes and approximately forty-four shared UI components.
  • The service tier uses FastAPI on Uvicorn with fifty-one REST endpoints across nine routers, plus a WebSocket channel for streaming optimization state to the browser.
  • Model binaries reside under ml-optimization saved_models, with shared preprocessing so that training code and inference code stay consistent — which was important for my reproducibility requirement.
  • PostgreSQL sixteen hosts the medallion schemas; the slide lists sixteen Bronze, sixteen Silver, and eight Gold tables for this project’s warehouse.
  • I use the ml_optimization schema for query logs, recommendation history, and job metadata that feed both features and monitoring views.
  • At a high level the server path is: validate input, extract features, run the four models, fuse scores, and push updates to connected clients.

SLIDE 9 — DATABASE DESIGN (BRONZE ERD)
────────────────────────────────────────────────────────────────────────────────
  • I briefly discuss Bronze-layer design because clean landing data supports trustworthy downstream analytics and telemetry.
  • Bronze holds normalized, OLTP-oriented entities prior to Silver conformity and Gold analytical tables.
  • The entity-relationship diagram emphasizes referential integrity — approximately twenty-one foreign keys — which reduces orphan rows before transformation jobs run.
  • I modeled person and location through an explicit many-to-many pattern rather than denormalizing duplicate persons.
  • restricted_info is a one-to-one extension of person for sensitive attributes, which follows a standard pattern for separating PII while preserving a single subject key.
  • employment uses a self-referencing foreign key for managerial hierarchy without redundant employee rows.
  • Every Bronze table includes source system, load timestamp, and batch identifier fields for auditability, as required in my design specification.

SLIDE 10 — EVALUATION RESULTS
────────────────────────────────────────────────────────────────────────────────
  • I will summarize the quantitative evaluation I report in Chapter — or section — on experiments.
  • All regressors share the same featurization and held-out partition so that improvements are comparable.
  • XGBoost exceeds Gradient Boosting by three point seven percentage points and Random Forest by six point six points on that split.
  • For Isolation Forest, using a thresholded evaluation protocol, I obtained approximately ninety-one point two percent test accuracy.
  • For latency prediction, roughly ninety percent of cases fall within plus or minus twenty percent error, with residual sigma near thirty-eight milliseconds.
  • I experimented with balanced sampling; it reduced overall accuracy by one point five three percentage points, so I document imbalanced training as the stronger configuration for my stated objectives.

SLIDE 11 — CASE STUDY (HEALTHCARE.GOV)
────────────────────────────────────────────────────────────────────────────────
  • I include a brief case study to illustrate capacity risk in production systems; I treat it as technical background rather than political commentary.
  • The slide references the October twenty thirteen launch period and PostgreSQL’s role in part of the architecture.
  • Planned capacity and observed traffic diverged sharply — millions of visitors versus tens of thousands of concurrent users assumed in planning.
  • At peak dysfunction only six registration completions could succeed simultaneously, indicating severe contention beyond a single slow statement.
  • The slide notes prolonged outage duration and remediation costs exceeding one point seven billion dollars at the program level.
  • Reported causes include unoptimized queries, lock contention, and insufficient tooling to prioritize fixes — which motivates telemetry-driven ranking.
  • I connect this to my project only modestly: my system does not prevent all outages, but it can help surface SQL-level risk earlier than ad hoc review alone.

SLIDE 12 — ALGORITHMIC FLOW
────────────────────────────────────────────────────────────────────────────────
  • The next slide shows the end-to-end pipeline I implemented.
  • Telemetry enters ml_optimization.query_logs in a manner consistent with pg_stat_statements snapshots.
  • I construct a fixed-length feature vector per row — thirteen dimensions on this slide, with log transforms as noted; my written thesis may list fifteen after lexical expansions, which I can clarify if you wish.
