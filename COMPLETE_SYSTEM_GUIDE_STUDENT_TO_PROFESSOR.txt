PART A — ARCHITECTURE AND DATA FLOW (CONTEXT FOR THE WHOLE PROJECT)
================================================================================

What I built at a systems level
--------------------------------------------------------------------------------
I implemented a three-tier pattern you would recognize from enterprise analytics
projects. The browser runs a React dashboard that speaks HTTP JSON and WebSockets
to a FastAPI service. That service is the only component allowed to connect to
PostgreSQL with credentials suitable for reading catalogs, monitoring schemas, and
ml_optimization tables. The models themselves live as files on disk next to the API
code; training happens offline via Python scripts, and inference happens inside route
handlers when a page asks for clusters, cache probabilities, anomaly scores, or
latency-ranked recommendations.

Why I insist on this separation
--------------------------------------------------------------------------------
If the dashboard queried PostgreSQL directly from the browser, I would expose
connection strings and bypass centralized validation. By forcing every UI through
/api/v1, I can attach metadata such as data_watermark_utc, window_start_utc,
window_end_utc, degraded_mode, and degraded_reason so evaluators know whether a chart
represents a full SQL rollup or a partial fallback.

Machine learning versus deterministic UI logic (definitions I use consistently)
--------------------------------------------------------------------------------
When I write “model inference,” I mean a saved artifact produced in training—examples
include WorkloadClusterer, QueryTimePredictor, CachePredictor, QueryAnomalyDetector—
applied to new rows at request time.

When I write “deterministic analytics,” I mean formulas in TypeScript that aggregate
already-logged statistics—for example ranking queries by total execution time or
computing trend percent between a one-day average latency and a long-window baseline.
Those formulas do not train weights from data at render time; they only combine numbers
the API already returned.

That distinction matters for academic integrity: my Analytics “ML hotspots” panel uses
deterministic analytics so I can defend the formula in closed form during questions.

Content name: Environment variables and API client (dashboard/src/services/api.ts)
--------------------------------------------------------------------------------
What it does in detail
  The module defines API_BASE from import.meta.env.VITE_API_BASE_URL, defaulting to
  http://localhost:8000/api/v1. Every REST path I document elsewhere is appended to
  that base. For WebSockets I derive ws:// or wss:// from the same host via
  getMlApiWebSocketOrigin(), unless VITE_WS_BASE_URL overrides it when the socket is
  proxied differently from REST.

How data moves through it
  request<T>() wraps fetch with JSON headers, no-store caching to avoid stale seminar
  demos, and unified parsing of FastAPI error payloads into ApiError with a detail
  string suitable for banners.

Internal logic worth mentioning
  I classify network failures (TypeError, connection refused patterns) and replace
  them with a short tutorial message that mentions starting start_services.py and
  setting VITE_API_BASE_URL—so a grader does not assume my code is broken when only
  the backend is stopped.

Why I centralized this
  Consistency across hooks: useDashboardData, useMonitoringData, useAnalyticsData,
  useAlertsData, and optimization apply all share the same error semantics.

Machine learning
  None in this file.

What appears on screen
  Users rarely see “api.ts”; they see the outcome through fewer mysterious failures and
  clearer error banners.

Hook name: useDashboardData
--------------------------------------------------------------------------------
Purpose
  Supplies DashboardPage with summary, sales, customers, alerts slice, health.

Fetch sequence in detail
  (1) Try GET /warehouse/home-dashboard once per deduped promise loadHomeDashboardOnce.
  (2) On failure other than 404, bubble error.
  (3) On 404 from bundle, parallel settle: getWarehouseSummary, getSalesStats,
      getCustomerStats, getActiveAlerts, getHealth.
  (4) Poll every useDashboardRefreshIntervalMs from monitoring preferences; silent
      refresh avoids flipping the initial loading flag.

Logic inside the hook
  If summary, sales, and customers are all null after success, I still set a generic
  failure message so you do not see an empty page silently.

Why I implemented polling
  During a live demo the warehouse ingests rows; refresh proves the UI is not static.


Hook name: useMonitoringData
--------------------------------------------------------------------------------
Purpose
  Single object merging jobs, definitions, pipeline, errors, throughput, freshness,
  dataQuality, failedRunsTotal, mlAnomalyCount.

Fetch sequence in detail
  (1) Parallel: GET /monitoring/etl/monitoring-page and GET /alerts/anomalies.
  (2) Parse anomalies total into mlAnomalyCount for ETLStats alignment with Alerts page.
  (3) If monitoring-page succeeds, merge all fields including failedRunsTotal via
      explicit keys—not errors.length.
  (4) If monitoring-page missing (404), try dashboard-bundle plus separate freshness and
      data-quality calls.
  (5) If still failing, legacy seven-call parallel path.

Logic inside the hook
  Generation counter cancels stale merges when user navigates quickly.

Why the staged fallback
  Older deployments might not ship the bundled route; I refuse to leave the page blank.


Hook name: useAnalyticsData
--------------------------------------------------------------------------------
Purpose
  Populates AnalyticsPage with three query-performance windows, rollups, hourly UTC
  series, optimization history and recommendations slices, optional storage and cost.

Fetch sequence in detail

  (2) Validate three performance payloads exist; throw readable contract error if not.
  (3) Parse metadata into contractMeta for watermark and degraded flags.
  (4) Async aux with fifteen-second timeout: getStorageUtilization, getCostTracking—

  (5) Fallback: getAnalyticsPageBundle plus three getQueryPerformance windows computed in
      UTC via utcPerformanceDateRange.

Why bundles first
  Single server-side transaction reduces drift between charts that must match the same
  SQL rollups.

Fetch sequence in detail

PART D — SYNTHESIS FOR ORAL DEFENSE (DETAILED)
================================================================================
End-to-end prediction story I would give if you asked for five minutes
--------------------------------------------------------------------------------
Telemetry originates from PostgreSQL execution: pg_stat_statements snapshots or equivalent
pipelines roll into ml_optimization.query_logs with timestamps and hashed identifiers.
Offline I train models using scripts under scripts/ml-optimization and ml-optimization/models:
WorkloadClusterer clusters queries into behaviors; QueryTimePredictor estimates latency impact;
CachePredictor estimates probability of cache benefit; QueryAnomalyDetector learns normal
regions in metric space with IsolationForest to flag outliers without labels.

Online, FastAPI routes load artifacts lazily, guard against missing files with degraded JSON,
and fuse model scores with SQL heuristics—candidate indexes must survive catalog checks before
I show Implement. The React layer badges recommendation provenance, renders cluster charts and
cache probabilities where endpoints succeed, and labels deterministic hotspot math separately so
my claims stay precise.

Limitations I would state explicitly
--------------------------------------------------------------------------------
Models reflect historical workloads—under drift they require retraining. Anomaly scores are not
classification accuracy in the supervised sense. Recommendations are advisory; economic optimality
still requires human judgment. Query text may be sensitive; access control belongs at deployment
time.

PART E — MACHINE LEARNING MODELS, RECOMMENDATIONS, AND THE UI (APPENDIX)
================================================================================
Professor, this closing appendix answers specifically how ML models participate in optimization
recommendations, where those recommendations surface in the React app, and how the Optimizations
page differs from the Analytics page with respect to recommendation payloads.

Important naming note (so there is no confusion)
--------------------------------------------------------------------------------
In my router (dashboard/src/App.tsx) there is no separate route called “Recommendations.” Index
and partition recommendations appear as the large tabbed card titled “Recommendations” on the
Performance Optimization Center — route /optimizations. The Analytics page (route /analytics)
loads a recommendations array inside GET /optimization/analytics-dashboard-bundle for API
completeness and future use, but the current AnalyticsPage.tsx does not render a recommendation
table; it focuses on stats, workload clusters, cache candidates, hotspots, slow-query impact, and
busy-hour charts. If you say “recommendations UI,” in this codebase that primarily means the
Optimizations page panels IndexRecommendations and PartitionRecommendations.


E.1 — Inventory of ML artifacts and where inference runs
--------------------------------------------------------------------------------
QueryTimePredictor (latency regression; often XGBoost JSON or pickle export)
  Loaded in optimization_routes when building live recommendations. Used to score candidate
  table-column pairs derived from workload aggregation so estimated_improvement and priority align
  with predicted execution impact.

QueryAnomalyDetector (Isolation Forest on engineered metrics)
  Optional partner to the predictor when scoring pairs or flagging severity proxies inside the
  recommendation builder.

WorkloadClusterer (KMeans or DBSCAN after scaling and optional PCA)
  Not used to paint each row of the IndexRecommendations list directly; it powers GET
  /optimization/workload-clusters on the Analytics “Smart insights” panel. Indirectly, workload
  segmentation informs how I explain heterogeneous traffic in the thesis even though the
  recommendation list is driven mainly by query_logs aggregates and catalog fusion.

CachePredictor
  Powers GET /optimization/cache-candidates — cache_probability rankings appear under Analytics Smart
  insights, not inside the index/partition recommendation tabs.

Train versus serve
  Training happens offline (scripts under scripts/ml-optimization and ml-optimization/models).
  Serving happens inside FastAPI when routes call _load_trained_models() and subsequent helpers.


E.2 — Backend: how a recommendation object is produced (core builder)
--------------------------------------------------------------------------------
The single internal function _build_optimization_recommendations_payload in
ml-optimization/api/routes/optimization_routes.py merges multiple sources into one JSON list each
row roughly containing: recommendation_id, type (index or partition), table, columns, sql_statement,
estimated_improvement, priority, status, query_count, avg_execution_time_ms, reason or explanation,
and recommendation_source (for example ml_query_logs, ml_pg_stat, ml_mixed, persisted_db,
pg_stat_heuristic).

Path A — Live ML branch (when analytics_bundle_fast is false and models exist)
  _generate_model_based_recommendations reads recent ml_optimization.query_logs under configured
  lookback and sample limits (environment variables such as OPTIMIZATION_QUERY_LOG_LIMIT and
  OPTIMIZATION_QUERY_LOG_LOOKBACK_HOURS appear in the GET /recommendations docstring). It groups
  workload evidence, invokes predictor and detector scoring where applicable, assigns priority,
  generates SQL text for suggested indexes or partitions, and tags recommendation_source from how
  evidence was blended.

Path B — Augmentation without skipping everything
  When not in fast bundle mode, code augments partition hints from workload tables and merges
  pg_stat_user_tables live hints when OPTIMIZATION_MERGE_PG_STAT_LIVE style configuration allows,
  so sequential scan pressure can surface in the same merged list.

Path C — Persisted rows
  Rows stored in ml_optimization.index_recommendations are SELECTed and merged so the UI shows both
  freshly generated candidates and durable records with status pending, applied, or rejected.

Path D — Catalog filtering
  _filter_genuine_recommendations removes suggestions that cannot reference real objects so I do
  not display DDL against nonexistent tables.

Sorting and caps
  Recommendations sort by priority and activity proxies; limits apply after merge (REST query
  parameter limit; WebSocket uses a capped slice for index and partition separately).


E.3 — Critical difference: full recommendations versus Analytics bundle “fast” recommendations
--------------------------------------------------------------------------------
GET /optimization/recommendations (used by Optimizations polling fallback and prefetch) calls
_build_optimization_recommendations_payload with analytics_bundle_fast defaulting to false. That
means live ML sampling and pg_stat merge stages run when models and flags permit — this is the
heavier, most current recommendation path.

GET /optimization/analytics-dashboard-bundle (used by useAnalyticsData for the Analytics page)
calls the same builder but passes analytics_bundle_fast=True for its recommendations slice. The
inline docstring states that fast mode skips live ML sampling, pg_stat merge, partition augmentation,
and per-row model rescore — it favors persisted merge and lighter work so the Analytics bundle can
complete alongside expensive triple query-performance SQL in parallel threads. Practically: the
Optimization Center shows the freshest merged recommendation pipeline under normal REST/WebSocket
calls; the Analytics bundle still carries a recommendations object for consistency and hooks, but
it is not the same computational path as the live optimizer GET.


E.4 — How recommendations reach the Optimizations page UI (route /optimizations)
--------------------------------------------------------------------------------
Transport sequence
  useOptimizationsData merges two concerns: (1) useOptimizationRealtimeWebSocket connects to
  WS /api/v1/ws/optimization-stream. Each tick the server builds an optimization_snapshot JSON
  containing index recommendations, partition recommendations, optimization history, and a
  performance snapshot for the socket’s fixed UTC window. The server implementation uses
  _build_optimization_snapshot_sync in websocket_routes.py, which calls
  _build_optimization_recommendations_payload with status_filter pending and splits rows into index
  versus partition arrays. (2) Immediately on mount the hook also prefetches via HTTP
  GET /optimization/recommendations so rows appear before the first WebSocket frame. (3)
  useQueryPerformanceFetch separately calls GET /optimization/query-performance for the user-selected
  Last 7/30/90 days window — that panel is not the ML recommendation list; it is telemetry ranked by
  total time.

Rendering sequence
  OptimizationsPage counts index versus partition rows from data.recommendations for KPI cards; the
  tabbed card switches between IndexRecommendations and PartitionRecommendations.

What each recommendation row suggests to the user
  Table and column targets; priority and estimated improvement framed as a fraction; textual reason;
  SQL preview; optional recommendation_source badge mapped in IndexRecommendations.tsx (examples:
  Live activity for ml_query_logs, Live database for ml_pg_stat, Live blended for ml_mixed, Saved
  record for persisted_db, System insight for pg_stat_heuristic).

How Implement ties back to the backend
  Clicking Implement posts POST /optimization/recommendations/{id}/apply with JSON containing
  optimization_id and optional snapshot from recommendationApplySnapshot for auditing. HTTP 409 is
  treated as already satisfied — the row dismisses and Optimization History records the outcome.


E.5 — Analytics page (route /analytics) and recommendations plus other ML views
--------------------------------------------------------------------------------
Recommendations array in the bundle
  useAnalyticsData stores data.recommendations from the analytics-dashboard-bundle response.
  Present-day AnalyticsPage.tsx does not map that array into a visible list component. If you open
  developer tools you still see recommendations inside the hook state for debugging or future UI.

Where ML actually appears on Analytics instead
  WorkloadCacheMlPanels — workload clusters from WorkloadClusterer assignment over query_logs sample;
  drill-down pages GET /optimization/workload-clusters/{cluster_id}/queries. Cache candidates from
  CachePredictor via GET /optimization/cache-candidates.

  MLHotspotTimeline — deterministic trend scoring across latency windows, not a fifth neural net.

  QueryPerformanceImpact and AnalyticsStats — aggregates from query performance slices and rollups.

So when you ask how ML “recommends data” to Analytics: clusters and cache probabilities are the
explicit ML-ranked outputs on that page; index or partition DDL suggestions remain primarily on the
Optimization Center where Implement lives.


E.6 — Oral summary you can quote
--------------------------------------------------------------------------------
Workload telemetry lands in ml_optimization.query_logs. Offline training produces artifacts.
Online, _build_optimization_recommendations_payload fuses live ML scoring when enabled, pg_stat and
workload hints, and persisted rows, then filters against the catalog. The Optimizations page consumes
that payload through WebSocket snapshots and GET /recommendations, rendering actionable index and
partition cards with provenance badges and Implement. The Analytics bundle includes a lighter
recommendations merge for bundle performance; the Analytics UI emphasizes clustering and cache
probability panels plus deterministic hotspot and slow-query analytics rather than a recommendation
table in the current build.

ML — MODELS AND FEATURES BY UI AREA (WHAT IS LEARNED VERSUS WHAT IS PURE SQL OR MATH)
================================================================================
Professor, this section maps each Analytics or Optimizations concept you asked about to the
actual artifact or algorithm where one exists, and explicitly calls out places where there is
no trained model so my thesis language stays precise.

--------------------------------------------------------------------------------
1. Index and partition recommendations (Optimization Center — route /optimizations)
--------------------------------------------------------------------------------
Which models
  QueryTimePredictor — primary supervised model for estimating latency-related benefit on candidate
  index or partition keys; implementation loads XGBoost JSON export or pickle depending on how I
  exported artifacts (see optimization_routes model loader).
  QueryAnomalyDetector — sklearn Isolation Forest; used as an auxiliary scorer when ranking table
  and column pairs so unusually harsh latency patterns influence priority together with predicted
  improvement.

Core builder
  ml-optimization/api/routes/optimization_routes.py defines _generate_model_based_recommendations
  and _model_score_for_table_column. The pipeline samples recent ml_optimization.query_logs under
  configurable lookback and row limits, aggregates workload evidence by table and column groups,
  invokes predictor and detector scoring, assigns priority (high, medium, low), fills estimated
  improvement and textual reason, emits provisional DDL in sql_statement, and tags
  recommendation_source (for example ml_query_logs, ml_pg_stat, ml_mixed).

Algorithms in plain terms
  Regression trees (XGBoost-style) estimate how much runtime might drop if an index or partition
  targets the predicates seen in logs.
  Isolation Forest treats engineered metric vectors as coordinates; points far from the bulk of
  normal traffic receive more anomalous scores, which I fold into severity-style ranking rather than
  treating them as classification labels.

Non-ML fusion you must still mention
  Current pg_stat_user_tables hints may merge into the same list when configured. Persisted rows in
  ml_optimization.index_recommendations merge with live rows. _filter_genuine_recommendations drops
  suggestions that fail catalog checks.

Partition versus index
  The same generator emits both types; partition rows carry partition_column and type partition.
  Augmentation helpers can add partition hints from workload table names.

--------------------------------------------------------------------------------
2. Query groups — workload clustering (Analytics → Smart insights — WorkloadCacheMlPanels)
--------------------------------------------------------------------------------
Which model
  WorkloadClusterer in ml-optimization/models/workload_clustering.py.

Algorithm
  Features such as normalized execution time, log row counts, join complexity and filter proxies are
  assembled into a matrix; StandardScaler fits on the training distribution; optional PCA when the
  feature width exceeds ten columns; then sklearn.cluster.KMeans or sklearn.cluster.DBSCAN depending on
  WorkloadClusteringConfig (algorithm, n_clusters, eps, min_samples).

How it assigns groups at serve time
  GET /optimization/workload-clusters loads the pickled clusterer, pulls a fresh dataframe slice from
  query_logs, calls predict_from_query_logs to assign cluster_id per row, aggregates cluster_counts and
  numeric profiles, returns exemplar query previews. Drill-down GET
  /optimization/workload-clusters/{cluster_id}/queries paginates rows belonging to one cluster after
  sorting by average latency within the cluster.

UI nuance
  The label “Largest group share” on that panel refers to the biggest cluster by row count among
  clustered queries — it is not the “Largest tables” bar list lower on the page.

Related ML on the same Smart insights row (cache column when enabled in build)
  CachePredictor (trained classifier in ml-optimization/models/cache_predictor.py) scores query
  templates for cache-benefit probability. GET /optimization/cache-candidates returns ranked rows with
  cache_probability — supervised learning from buffer-related features in training data, not the same
  algorithm as workload clustering.

--------------------------------------------------------------------------------
3. ML Hotspot Timeline (Analytics — ML hotspots column beside Smart insights)
--------------------------------------------------------------------------------
Which model
  None. This panel is deterministic analytics implemented in dashboard/src/components/analytics/
  MLHotspotTimeline.tsx using utilities in analyticsDerived.ts.

How it works
  I union query identifiers across one-day, seven-day, and retention-long query-performance slices in a
  fixed priority order. For each fingerprint I compare average latency milliseconds across windows,
  compute trend percent relative to a baseline that prefers the long window then falls back to seven-day,
  and compute an impact-style score from executions multiplied by latency-like totals so frequent heavy
  queries rank higher than rare spikes.

Why I still label it “ML” in navigation
  It highlights hotspots operators care about in ML-assisted tuning workflows even though the ranking
  formula itself is explicit arithmetic, not gradient descent.

--------------------------------------------------------------------------------
4. Slow query impact (Analytics — Slow query impact card)
--------------------------------------------------------------------------------
Which model
  None for the ranking itself.

How it works
  QueryPerformanceImpact reads the seven-day normalized query-performance array from the analytics
  bundle. deriveQueryAggregates computes totals and slow share; topQueriesByTotalTime selects the top
  eight fingerprints by summed wall-clock time in that window. Impact percentage bars divide each row’s
  total time by the sum across those eight displayed rows so the bars sum to one hundred percent among the
  visible set.

Data origin
  Server-side aggregation over ml_optimization.query_logs or equivalent statistics exposed by GET
  /optimization/query-performance — not a neural network scoring pass in the browser.

--------------------------------------------------------------------------------
5. Busy times (Analytics — Busy times half of “Busy times and largest tables”; also peak hour in overview stats)
--------------------------------------------------------------------------------
Which model
  None.

How it works
  Prefer hourlyCallsUtc7d from the analytics-dashboard-bundle SQL rollup — twenty-four floats for UTC
  hour totals over seven days when available. WorkloadPatterns converts UTC counts into local clock
  buckets for display, then rollupHourlyExecutions merges hours according to Settings → Metrics
  aggregation (for example one-hour versus wider buckets). If hourly series is missing, the component
  derives hourly shapes from seven-day query-performance rows via deriveWorkloadHourlyExecutions.

Peak detection
  peakFromRolledBuckets scans rolled bucket values for maximum calls — classical argmax, not clustering.

--------------------------------------------------------------------------------
6. Largest tables (Analytics — bottom list under “Busy times and largest tables”)
--------------------------------------------------------------------------------
Which model
  None.

How it works
  WorkloadPatterns calls topTablesFromStorage(data.storageUtilization, 6). That helper parses GET
  /storage/utilization JSON — per-schema utilization lists — sorts tables by on-disk size in megabytes,
  and shows the top six with horizontal bars scaled to the largest among them. Sizes come from PostgreSQL
  catalog and storage statistics computed server-side, not from workload clustering.

If storage fails to load
  The UI prints that storage data is unavailable — the hook may time out storage aux calls without
  blocking core analytics.

--------------------------------------------------------------------------------
7. Anomaly detection (Alerts → Anomalies tab; related counts on Monitoring ETL stats row)
--------------------------------------------------------------------------------
Which models and rules
  Primary ML path: QueryAnomalyDetector using sklearn.ensemble.IsolationForest on vectors built from
  query_logs fields such as mean_exec_time_ms, calls, rows_affected, shared buffer hits and reads.
  detect_anomaly returns whether the point is flagged, a continuous score, and a textual reason such as
  execution time spike. Severity tiers derive from score magnitude and reason strings inside
  alert_routes._detect_model_anomalies.

Heuristic paths (not Isolation Forest)
  Failed ETL runs from monitoring.job_runs in the last seven days.
  Insert-rate anomalies via pg_stat_user_tables comparing n_tup_ins across bronze, silver, gold tables.
  Additional structural heuristics documented in alert_routes.py (for example unusual row size).

How results reach UI
  GET /alerts/anomalies or bundled alerts page-bundle returns JSON rows with type model_anomaly versus
  heuristic types; AlertsPage formats messages and optional actual versus expected fields.
